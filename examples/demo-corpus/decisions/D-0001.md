# Cache invalidation via fixed 5-minute TTL

## Context / Problem
Product listing queries hit the database on every request. Response times degraded as traffic grew, and most reads returned identical data.

## Decision
Cache product listings in Redis with a fixed 5-minute TTL; no explicit invalidation.

## Rationale
Simplest possible policy: no invalidation hooks in the write path, and a bounded staleness window that product owners accepted at the time.

## Outcome / Learned
Worked until flash-sale pricing made 5-minute staleness unacceptable. Superseded by event-driven invalidation (D-0002).
